# 企業・法人辞書とロゴ取得の方針

Job Hunt Vault では、ユーザーが管理する選考データと、外部由来の企業参照データを分けて扱います。

## 現在の収録状態

D1 に公式データを入れる前でも、代表的な企業・官公庁・法人を探せるように、Worker には小さな内蔵辞書を持たせています。

- 内蔵辞書: 117件
- 用途: 初回入力の補助、D1辞書が空のときのフォールバック
- 例: `東京メトロ` のような通称から `東京地下鉄株式会社` を探せるよう、別名も検索対象に含めます。

これは国内網羅用の本体ではありません。国内の上場企業を広く扱う場合はJPX、非上場企業・合同会社・国立研究開発法人・独立行政法人・省庁・自治体まで広げる場合は国税庁法人番号データをD1へ取り込む前提です。

## 1. ユーザーの選考データ

`companies` はルーム内で実際に追跡する選考先です。

- 企業名
- 読み
- ドメイン
- 業種
- 直近締切
- 証券コード
- 取引所
- 採用ページURL
- MyPage URL
- ロゴURL
- メモ

ここには「自分が今選考管理している企業・法人」だけを置きます。

## 2. 参照用の企業辞書

`company_catalog` は企業検索や入力補完に使う辞書です。

- source: `jpx`, `manual`, `logo.dev`, `custom` など
- source_id: 公式データ側のID
- country
- name
- name_kana
- normalized_name
- domain
- industry
- market
- ticker
- exchange
- logo_url
- metadata_json

JPX の上場会社一覧、証券コード、市場区分、業種などは、この `company_catalog` に取り込みます。選考データとは別なので、後から辞書を更新してもユーザーの進捗やメモには触りません。ユーザーが選考先一覧に追加するときだけ、企業名、読み、業種、証券コード、取引所、候補ドメインを `companies` 側へコピーします。

`domain` は公式サイトやロゴ解決のための候補です。採用ページURLや、ログイン後に使うMyPage URLとは別物なので、辞書反映時にMyPage URLへ自動コピーしません。採用ページURLとMyPage URLは、実際の応募導線を確認してから手入力します。

## 3. JPXなどを入れる理由

日本企業を大量に扱う場合、毎回企業名・業種・証券コードを手入力すると破綻します。公式データを辞書として取り込むと、次のことができます。

- 企業名の補完
- 50音順に近い並べ替え
- 業種別の絞り込み
- 上場/非上場の区別
- 証券コードや市場区分の表示
- ロゴ取得に使う候補ドメインの整理

ただし、JPXだけでは非上場企業、合同会社、官公庁、自治体、国立大学法人、独立行政法人、Webサイトドメイン、ロゴまでは揃いません。ロゴは Logo.dev などのprovider、非上場・公的機関は国税庁法人番号データ、ユーザー入力、外部provider、手動補完を併用します。

## 4. 国内網羅に近づけるデータ源

「国内のほぼすべて」を狙う場合は、次の順で広げます。

1. JPXの東証上場銘柄一覧
   - 上場企業、証券コード、市場区分、業種の土台にします。
   - プライム、スタンダード、グロースなどの市場区分を埋めます。
2. 国税庁法人番号公表サイトの基本3情報
   - 商号又は名称、所在地、法人番号を持つ法人等を広く補完します。
   - 国の機関、地方公共団体、株式会社、有限会社、合名・合資・合同会社、その他の設立登記法人などを法人区分へ寄せます。
3. 手動CSV
   - 採用マイページURL、候補ドメイン、略称、読み、業種の補正に使います。
   - 公式データにない「東京メトロ」「JR東日本」のような通称は `aliases` 列で補います。

法人番号データは件数が非常に大きいため、Workerに静的同梱せず、D1へ段階投入するか、必要な法人だけ抽出して入れる運用にします。数千件はJPX、数万件は国税庁データの都道府県単位CSVから始め、全国全件は動作確認後に分割投入します。

## 5. ロゴ取得

ロゴ画像はリポジトリに同梱しません。

実装方針:

- `GET /api/logo/resolve?domain=example.com`
  - ドメインからLogo.dev画像URLを作り、`logo_cache` に保存します。
- `GET /api/logo/search?q=sony`
  - 企業名からLogo.devの候補を検索します。
  - Word/Officeのオンライン画像検索に近い体験ですが、無断スクレイピングではなくprovider APIを使います。

キーの扱い:

- `LOGO_DEV_PUBLISHABLE_KEY`
  - 画像CDN用のpublishable keyです。
  - Cloudflareの環境変数またはsecretに置きます。
- `LOGO_DEV_SECRET_KEY`
  - 企業名検索用のsecret keyです。
  - 必ずCloudflare Workers Secretsに置きます。

チャットやGitHubに貼らないもの:

- `LOGO_DEV_SECRET_KEY`
- `.env`
- `.dev.vars`
- provider dashboardのprivate token

チャットに貼ってよいもの:

- 「Logo.devのkeyを登録済み」
- 「検索用secretは未登録」
- provider名

## 6. インポート運用

最初は空の `company_catalog` で問題ありません。CloudflareのD1ができた後に、次の段階で取り込みます。

1. JPXなどの公式データを作業用フォルダへ置く
2. import scriptで `company_catalog` 用のSQLへ変換する
3. D1に投入する
4. アプリ上で企業名検索、業種絞り込み、締切順/50音順/業種順を使う

CSVからSQLへ変換する例:

```bash
npm run catalog:sql -- --input ./work/jpx.csv --out ./work/company_catalog.sql
```

このscriptは、JPXや手動CSVの列名をいくつかの別名で受けます。最低限、企業名と証券コードがあると安定します。業種、市場区分、取引所、読み、ドメインがある場合は `company_catalog` に一緒に入れます。

別名・略称を入れる例:

```csv
証券コード,会社名,読み,業種,市場区分,取引所,ホームページ,別名
9023,東京地下鉄株式会社,とうきょうちかてつ,陸運・鉄道,プライム,TSE,tokyometro.jp,東京メトロ;tokyo metro
```

`別名`、`略称`、`通称`、`aliases`、`alias`、`short_name`、`common_name` は検索用の `normalized_name` に入ります。保存される正式名称は `会社名` 側です。

国税庁法人番号CSVを分割SQLへ変換する例:

```bash
npm run catalog:sql -- --preset nta --source nta --input ./work/nta_tokyo.csv --out-dir ./work/catalog-sql --chunk-rows 5000 --active-only
```

`--preset nta` はヘッダーなしの法人番号CSVにも対応します。`法人番号`、商号又は名称、フリガナ、法人種別、都道府県、市区町村、郵便番号、英語表記などを `metadata_json` と検索用文字列へ寄せます。`--active-only` を付けると、閉鎖済みまたは最新履歴ではない行を可能な範囲で除外します。

大量投入の順番:

1. JPX上場銘柄を入れる
2. 国税庁データを東京都など1都道府県だけで試す
3. 検索速度と候補の見え方を確認する
4. 必要な地域を追加する
5. 全国全件は、D1の容量と検索速度を確認してから分割投入する

public repo には、secret、実ユーザーデータ、DB dump、商用ロゴ画像そのものを入れません。
